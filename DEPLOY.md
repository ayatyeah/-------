# Публикация на Railway

Сайт готов к запуску: сервер сам отдаёт и API, и собранный фронтенд на одном
порту, поэтому на Railway нужен **один сервис**, а не два.

Railway сам увидит `package.json`, выполнит `npm install`, `npm run build`
и запустит `npm start`. Отдельная команда сборки не нужна.

---

## 1. Обязательные переменные окружения

Задаются в Railway → **Variables**. Без них сайт поднимется, но будет дырявым.

| Переменная | Значение | Зачем |
|---|---|---|
| `PORT` | `8080` | **Критично.** Без неё сайт отвечает «Application failed to respond» — см. ниже |
| `ADMIN_PASSWORD` | свой длинный пароль | **Критично.** По умолчанию `admin` — на публичном адресе в админку зайдёт кто угодно и удалит каталог |
| `STORE_PATH` | `/data/store.json` | **Критично.** Иначе заявки клиентов пропадут при каждом деплое — см. пункт 2 |
| `JWT_SECRET` | длинная случайная строка | Подписывает сессии админки. Без неё вход слетает после каждого деплоя |
| `CORS_ORIGIN` | адрес сайта, например `https://cxmagro.up.railway.app` | Ограничивает, кто ходит в API |

`JWT_SECRET` удобно сгенерировать так:

```
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

Сессия админки живёт 12 часов (меняется переменной `SESSION_TTL_HOURS`), после
чего нужен повторный вход. Смена `ADMIN_PASSWORD` мгновенно обрывает все
активные сессии.

### Про `PORT` — здесь легко потерять полдня

Здесь раньше было написано «`PORT` задавать не нужно, Railway подставит его
сам». **Это неверно, и именно на этом деплой встал.**

Railway направляет входящий трафик на конкретный порт сервиса. Если
переменной `PORT` нет, сервер берёт запасной `3001` — и слушает не тот порт,
куда стучится платформа. Процесс при этом живой, в логах всё зелено, а
снаружи — «Application failed to respond». В логе это видно по строке:

```
✓ API СХМ Агро слушает 0.0.0.0:3001      ← 3001 = PORT не задан
```

Задайте `PORT=8080` в Variables — сервер послушно займёт его. (Годится любой
порт, важно лишь, чтобы Railway и приложение сошлись на одном; в Settings →
Networking у публичного домена должен стоять тот же.)

## 2. Данные не переживут перезапуск без диска

Базы данных у проекта сейчас нет: всё лежит в одном файле `data/store.json`.
Файловая система контейнера на Railway **временная** — она стирается при каждом
деплое и перезапуске. Без отдельного диска пропадут:

- заявки, которые оставили клиенты (самое дорогое),
- всё, что правили в админке: модели, новости, услуги, настройки.

Восстановятся только начальные данные из `server/seed.js`.

**Как чинится:** Railway → сервис → **Volumes** → создать диск с точкой
монтирования `/data`, затем в Variables прописать `STORE_PATH=/data/store.json`.
Код это уже поддерживает — менять ничего не нужно.

Диск обязателен, если собираетесь принимать реальные заявки.

---

## 3. Версия Node

В `package.json` закреплено `"engines": { "node": ">=22.9.0" }`, есть `.nvmrc`.

Это не формальность: `npm start` использует флаг `--env-file-if-exists`,
который появился только в Node 22.9. На Node 18 или 20 сервер упадёт на старте
с ошибкой `bad option`. Не понижайте версию.

## 4. ИИ (по желанию)

Без ключей анализатор лидов и чат работают на правилах — сайт полностью
функционален, ничего не ломается. Библиотек ставить не нужно: запросы идут
обычным `fetch`.

Порядок провайдеров: **OpenAI → Gemini → правила**. Первый, у кого есть ключ
и кто ответил, тот и отвечает.

В Railway → Variables добавьте:

| переменная | зачем |
|---|---|
| `OPENAI_API_KEY` | основной провайдер |
| `OPENAI_MODEL` | необязательно, по умолчанию `gpt-5-mini` |
| `GEMINI_API_KEY` | резерв, если OpenAI не ответил |
| `GEMINI_API_KEY_2`, `GEMINI_API_KEY_3` | необязательно: при лимите (429) сервер берёт следующий ключ |
| `GEMINI_MODEL` | необязательно, по умолчанию `gemini-2.5-flash` |
| `AI_ORDER` | необязательно: `gemini,openai` вернёт Gemini на первое место |

Ключи из локального `.env` на Railway **не попадают** — `.env` в git не
хранится. Их нужно завести в Variables отдельно, иначе на боевом сайте ИИ
молча уйдёт в правила, хотя локально всё работает.

Проверить, что получилось: `GET /api/ai/status` вернёт, какой провайдер
подхватился.

Ответы кэшируются в том же `store.json`, что и контент, — оценка заявки не
пересчитывается, пока сама заявка не изменилась. Отсюда следствие: **без
диска из шага 2 кэш теряется при каждом деплое** вместе с заявками, и за
разбор придётся платить заново.

## Короткий чек-лист

- [ ] Создать сервис из репозитория GitHub
- [ ] Variables → `PORT=8080`
- [ ] Volumes → диск на `/data`
- [ ] Variables → `STORE_PATH=/data/store.json`
- [ ] Variables → `ADMIN_PASSWORD=…` (не `admin`)
- [ ] Variables → `CORS_ORIGIN=https://ваш-домен`
- [ ] Variables → `OPENAI_API_KEY` (и `GEMINI_API_KEY` в резерв), если нужен ИИ
- [ ] Задеплоить и открыть сайт
- [ ] Открыть `/admin` и проверить вход
- [ ] Открыть `/api/ai/status` — убедиться, что движок не `rules`
- [ ] Оставить тестовую заявку → сделать redeploy → убедиться, что заявка на месте

Последний пункт — главная проверка: если заявка пропала, значит диск не
подключился и `STORE_PATH` смотрит не туда.

## Если что-то не так — сначала читайте лог

Сервер сам пишет о том, чего ему не хватает. В первых строках после запуска:

```
✓ API СХМ Агро слушает 0.0.0.0:8080
  Данные: /data/store.json
  ИИ: openai подключён
```

Так выглядит правильно настроенный сервис. Отклонения:

| в логе | что это значит |
|---|---|
| `слушает 0.0.0.0:3001` + `⚠ PORT не задан` | сайт снаружи не ответит — задайте `PORT` |
| `Данные: /app/data/store.json` + `⚠ STORE_PATH не задан` | диска нет, заявки сотрутся при деплое |
| `ИИ: правила (ключи не заданы)` | ключей нет в Variables; локальный `.env` на Railway не попадает — он не в git |
