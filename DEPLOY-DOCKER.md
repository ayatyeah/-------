# Запуск на своём сервере (DigitalOcean droplet)

Сайт целиком собирается в Docker-образ: один контейнер отдаёт и API, и
страницы. Рядом крутится второй — он раз в сутки складывает копию данных
на диск сервера.

Подходит любой Linux-сервер с Docker. Минимальный droplet (1 ГБ памяти)
справляется; при 512 МБ сборка образа может не хватить памяти — тогда
собирайте образ на своей машине и загружайте готовый.

---

## 1. Подготовить сервер

В панели DigitalOcean создайте droplet из образа **Docker on Ubuntu**
(Marketplace) — тогда Docker уже установлен. Если берёте чистую Ubuntu:

```bash
curl -fsSL https://get.docker.com | sh
```

Дальше заходим на сервер и забираем проект:

```bash
ssh root@ВАШ_IP
git clone https://github.com/ayatyeah/-------.git shm-agro
cd shm-agro
```

---

## 2. Задать пароли и ключи

```bash
cp .env.example .env
nano .env
```

Обязательно заполнить:

| Переменная | Что поставить |
|---|---|
| `ADMIN_PASSWORD` | свой длинный пароль. **Не `admin`** — иначе в админку войдёт кто угодно |
| `JWT_SECRET` | длинная случайная строка, команда ниже |
| `CORS_ORIGIN` | адрес сайта: `http://ВАШ_IP` или `https://ваш-домен` |
| `OPENAI_API_KEY` | если нужен ИИ (без него работают правила) |

Секрет для сессий:

```bash
openssl rand -hex 32
```

`PORT` и `STORE_PATH` в `.env` **не нужны** — их задаёт `docker-compose.yml`.

---

## 3. Запустить

```bash
docker compose up -d --build
```

Первая сборка занимает 1–3 минуты. Проверить:

```bash
docker compose ps          # оба контейнера должны быть Up, web — healthy
docker compose logs web    # тут же видны предупреждения о настройках
curl localhost/api/health  # {"ok":true,...}
```

Сайт открыт на `http://ВАШ_IP`, админка — `http://ВАШ_IP/admin`.

---

## 4. Если сайт упадёт — поднимется сам

За это отвечает `restart: unless-stopped` в `docker-compose.yml`. Он
покрывает три случая:

- **процесс упал с ошибкой** — Docker поднимает контейнер заново;
- **сервер перезагрузили** — контейнеры стартуют вместе с системой;
- **закончилась память, процесс убит** — тоже перезапуск.

При этом ручную остановку он уважает: после `docker compose stop` сам не
вернётся, пока не скажете `start`.

Отдельно сервер каждые 30 секунд проверяется изнутри (`/api/health`).
Состояние видно в `docker compose ps` — колонка `STATUS`, значение
`healthy` или `unhealthy`.

**Важная оговорка.** Docker перезапускает контейнер, когда процесс
*упал*. Если процесс жив, но завис и перестал отвечать, `unless-stopped`
его не тронет — метка `unhealthy` появится, а перезапуска не будет.
Чтобы перезапускать и такие случаи, добавьте сторожевой контейнер:

```yaml
  autoheal:
    image: willfarrell/autoheal
    restart: unless-stopped
    environment:
      AUTOHEAL_CONTAINER_LABEL: all
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
```

Учтите цену такого решения: доступ к `docker.sock` — это полные права над
Docker на сервере. Для сайта такого размера зависания маловероятны, поэтому
по умолчанию сторож не включён.

## 5. Бэкапы

Контейнер `backup` делает копию сразу при запуске и дальше раз в сутки.
Копии лежат прямо на сервере, в папке `backups` рядом с проектом:

```bash
ls -lh backups/
# store-2026-07-24_15-30.json.gz
```

Хранятся 14 дней, старые удаляются автоматически. Поменять срок — в
`docker-compose.yml`, переменная `BACKUP_KEEP_DAYS`.

Копия делается на ходу: сервер записывает данные атомарно, поэтому в архив
попадает либо целый новый файл, либо целый предыдущий. Останавливать сайт
не нужно. Каждый архив сразу проверяется на читаемость — битый удаляется, а
в логе появляется предупреждение.

Сделать копию прямо сейчас:

```bash
docker compose exec backup sh /scripts/backup.sh
```

Посмотреть историю:

```bash
docker compose logs backup
```

### Восстановление

Из самой свежей копии:

```bash
docker compose stop web
docker compose run --rm restore
docker compose start web
```

Из конкретной:

```bash
docker compose stop web
docker compose run --rm restore sh /scripts/restore.sh store-2026-07-20_03-00.json.gz
docker compose start web
```

**Сайт обязательно остановить.** Он держит данные в памяти и, продолжая
работать, перезапишет восстановленный файл своими — восстановление пропадёт
впустую.

Текущие данные при этом не теряются: перед заменой они откладываются рядом
под именем `store.json.before-restore-…`, на случай если вернули не ту копию.
Архив проверяется дважды — на целостность и на то, что внутри действительно
данные сайта; при любой нестыковке восстановление отменяется, ничего не
трогая.

### Копии за пределы сервера

Всё вышесказанное защищает от испорченных данных и неудачного обновления,
но **не** от потери самого droplet. Если сервер пропадёт, пропадут и копии
на нём. Раз в неделю забирайте архивы к себе:

```bash
scp root@ВАШ_IP:~/shm-agro/backups/*.gz ./
```

Или включите снапшоты droplet в панели DigitalOcean.

## 6. HTTPS

Пока сайт работает по `http://`, **пароль администратора идёт по сети
открытым текстом** — его может прочитать любой на пути. Для показа
заказчику это терпимо, для настоящей работы — нет.

Когда появится домен, направьте его A-записью на IP сервера и:

```bash
echo 'WEB_BIND=127.0.0.1:3001' >> .env
DOMAIN=ваш-домен.kz EMAIL=info@ваш-домен.kz docker compose --profile tls up -d --build
```

Caddy сам получит сертификат Let's Encrypt и будет продлевать его без
напоминаний. Строка `WEB_BIND` обязательна: она убирает сайт с внешнего 80
порта, который занимает Caddy, — иначе они подерутся за порт.

Не забудьте поменять `CORS_ORIGIN` в `.env` на `https://ваш-домен.kz`.

---

## 7. Обновление сайта

```bash
cd shm-agro
git pull
docker compose up -d --build
```

Данные и бэкапы не трогаются: они лежат на отдельном томе, а не внутри
образа.

---

## 8. Файрвол

Открыть нужно только то, чем пользуются:

```bash
ufw allow OpenSSH
ufw allow 80/tcp
ufw allow 443/tcp   # если включили HTTPS
ufw enable
```

## Что смотреть, когда что-то не так

```bash
docker compose ps                  # кто запущен и здоров ли
docker compose logs -f web         # что говорит сайт
docker compose logs backup         # делаются ли копии
docker stats --no-stream           # память и процессор
```

Сервер сам пишет в лог, чего ему не хватает. Например:

| В логе | Что значит |
|---|---|
| `⚠ ADMIN_PASSWORD = «admin»` | стандартный пароль, в админку войдёт любой |
| `⚠ JWT_SECRET не задан` | вход будет слетать при каждом перезапуске |
| `⚠ STORE_PATH не задан` | в Docker не должно появляться: путь задан в compose |
| `ИИ: правила (ключи не заданы)` | ключей нет, чат отвечает по ключевым словам |
